# Fix: Chrome Overlay Must Only Cover Pinned Headers

## Problem

`SwiftChromeHeader` registers with `visibleHeight = extent` always (not just when pinned). With 50 sections, the red overlay covers 50 * 44 = 2200px — the entire screen.

`SwiftChromeHeader` is a `RenderBox` (child of `SliverPinnedHeader`), so it can't use `SliverLayoutBuilder` to detect pinning.

## Solution

Replace the box-based `SwiftChromeHeader` + `SliverPinnedHeader` combo with a single custom sliver that handles BOTH pinning AND chrome registration.

### `SwiftPinnedHeaderChromeSliver` — new widget

A `SliverPersistentHeader(pinned: true)` that uses `SliverLayoutBuilder` internally to detect pinning and register with the chrome controller. Returns zero layout/paint extent — it's a pure detector that doesn't render visible content.

Pinning detection:
```
constraints.scrollOffset >= constraints.precedingScrollExtent → pinned
```

When pinned: registers with chrome controller (`visibleHeight = height`).
When not pinned: unregisters (`visibleHeight = 0`).

### Usage change

Before:
```dart
SliverPinnedHeader(
  child: SwiftChromeHeader(
    height: 44,
    child: Container(...),
  ),
)
```

After:
```dart
SwiftPinnedHeaderChromeSliver(height: 44),
SliverPinnedHeader(
  child: Container(height: 44, ...),
)
```

The chrome sliver is a separate sliver placed just before the `SliverPinnedHeader`. It detects pinning. The `SliverPinnedHeader` handles the actual visual pinning.
